1. Sample rate mismatch (48 kHz vs 44.1 kHz)
WASAPI render clients expect audio in the device’s mix format (often 48 kHz, stereo, float).

If you feed 44.1 kHz PCM directly, it plays back pitched up.

Fix:

Query the device mix format with IAudioClient::GetMixFormat.

Resample incoming PCM to match that format before writing.

Qt has QAudioResampler (or you can use libsamplerate / custom resampler).



2. Mono files distorted
Same root cause: channel mismatch.

If the device expects 2 channels and you feed 1 channel, WASAPI interprets the buffer incorrectly.

Fix:

Detect mono input and duplicate samples into stereo before writing.

Example: L = sample; R = sample; for each frame.



3. No parallel playback (sounds cut each other off)
Right now, you’re using a single IAudioClient for all SoundInstances.

That means only one render stream is active at a time. When a new sound starts, it reuses the same client and effectively resets playback.

Fix:

Each SoundInstance should create its own IAudioClient (via WasapiManager::activateDevice() or similar).

That way, multiple render clients can feed the same output device simultaneously.

Alternatively, implement a mixer: one render client, but mix all active sounds into a single buffer before writing. This is more efficient but requires mixing logic.



4. Stop delay (~1 second)
The delay comes from your playback loop sleeping (QThread::msleep(1)) while waiting for buffer space.

When you set stopFlag, the loop only checks it once per iteration, so it can take up to ~1 second to break.

Fix:

Check stopFlag immediately after GetBuffer() and before ReleaseBuffer().

Reduce sleep granularity (e.g., msleep(1) → QThread::usleep(100) or use event‑driven wait).

Or call IAudioClient::Stop() directly when stopping — this halts the stream immediately.
