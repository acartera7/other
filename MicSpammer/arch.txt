🛠️ Recommended Architecture
Mic worker thread

Captures mic input with IAudioCaptureClient.

Writes it into VB‑Cable’s render buffer continuously.

Runs until stopAllAudio is set.

Soundboard fire‑and‑forget threads

Each one loads a file, decodes PCM, and feeds it into VB‑Cable’s render buffer.

They don’t touch mic input at all.

WASAPI mixes them with the mic stream automatically.

Global stop flag

std::atomic<bool> stopAllAudio.

Checked in every loop (mic + soundboard).

When set, all threads exit gracefully.

What should be saved in a profile
Device selections

Input device (mic) ID

Output device ID

Monitoring device ID (if applicable)

Any other WASAPI endpoints you expose

Loaded files and references

Paths to audio files loaded in each page/module

Display names or aliases you show in the UI

Folder location for browsing/auto‑loading

User settings

Volume levels (mic passthrough, monitoring, playback)

Slider positions (e.g. outputVolume, monitoringVolume)

Any EQ or DSP settings if you add them later

UI state

Which page/tab was last active

Expanded/collapsed panels

Last selected profile itself

Global options

Whether monitoring is enabled

Whether mic passthrough is active

Latency/buffer size preferences

🔧 How to structure persistence
Profiles as JSON:
Save everything in a JSON file (device IDs, file paths, volumes, flags). Easy to serialize/deserialize.

Profile location:

Default: AppData\Roaming\YourApp\profiles.json

Allow user to export/import profiles to custom folders.

Startup behavior:

On launch, check if a “last used profile” exists.

Prompt user: “Load last profile (ProfileName)?” with Yes/No.

If Yes → restore devices, files, sliders, etc.

If No → start clean.
